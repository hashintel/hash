- Focused proof exists in:

  - `src/mastra/steps/discovery/extract-structured.test.ts`

Implications traced in chat:

1. The change is safely additive at the contract level. `linkedRosterEntryIds` is optional, so older persisted runs and existing fixtures still validate.
2. `rosterEntryId` remains the canonical subject-side anchor. `linkedRosterEntryIds` is a secondary participation index, not a new source of truth.
3. `hash-frontend` currently ignores the new field:
   - `apps/hash-frontend/src/pages/ingest.page/types.ts` does not declare `linkedRosterEntryIds`
   - `apps/hash-frontend/src/pages/ingest.page/results-panel.tsx` groups claims only by `claim.rosterEntryId`
4. Runtime is fine today, but the UI still will not show the same claim on the object entity's card until the frontend groups by `linkedRosterEntryIds`.
5. If the frontend starts using the new field, it must dedupe by `claimId` per entity card. The same claim may legitimately appear on multiple cards, but it must remain one claim identity.
6. The semantics are intentionally asymmetric across extraction paths right now:
   - prose path always emits `[subject]`
   - structured path may emit `[subject, object]` when the object exactly matches a roster entry
   Downstream should treat `linkedRosterEntryIds` as “known participants,” not “complete participants.”

```text
- ASSUMPTION: `rosterEntryId` remains the canonical subject-side link and `linkedRosterEntryIds` is only a secondary participation index → VALIDATE: preserve existing selection/evidence behavior keyed by `claimId` and keep grouping logic additive only.

```
